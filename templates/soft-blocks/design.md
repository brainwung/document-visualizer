# Soft Blocks（柔彩模块风）

一种和 `minimalist` 明显不同的模块化长图风格：不使用讲义式白底段落，也不依赖线框报表，而是用浅色背景、圆角块面、局部深色模块和柔彩分区来组织信息。适合数据复盘、文档总结、方案拆解、活动信息、产品说明、模块化报告和轻量 dashboard 型长图。

## When To Use

- 用户希望页面不是文章讲义，而是更像现代信息面板、模块化 dashboard 或 app 式内容看板。
- 源材料可以拆成多个相对独立的信息模块，例如核心结论、数据指标、时间线、流程步骤、角色分工、活动信息、方案要点、Top 列表或分类条形图。
- 需要比 `minimalist` 更活泼、更有色块分区，但仍要保持信息层级清楚、事实边界明确。

## Visual System

- 画布：1080px 宽，高度随内容展开。
- 外层背景：极浅蓝灰，推荐 `#E7EDF3`。
- 页面背景：近白冷色，推荐 `#F7FAFC`。
- 模块背景：以白色和浅色块面为主；深色高饱和背景只用于重点结论、结语、最终产出、关键警示或用户明确要求强调的核心模块。
- 主文字 / 浅色模块数字：深石墨色，推荐 `#2F3948`。
- 正文辅助：`#697482`。
- 弱辅助 / 英文装饰：`#9BA6B3`。
- 推荐色组：
  - 紫色主模块：`#6518F4`
  - 粉色强调模块：`#FF1F64`
  - 绿色行动 / 条形图：`#20CC72`
  - 青色趋势 / 柱状图：`#20B8E5`
  - 蓝色浅底：`#EAF1FF`
  - 粉色浅底：`#FFE8EF`
  - 绿色浅底：`#E7F9EF`
  - 青色浅底：`#E4F8FC`
  - 橙色浅底：`#FFF7E8`
- 模块圆角：18px。
- 模块不使用重投影；如果必须用阴影，只能非常轻，默认不用。
- 中文：AlibabaPuHuiTi 或系统现代无衬线。
- 数字：优先使用更圆润的现代无衬线（AlibabaPuHuiTi / system / Helvetica Neue），不强制 DIN。
- letter-spacing 保持 0。

## Structure

1. 顶部使用完整宽度的单栏标题区：大标题、说明文字。不要默认加入来源、类型、风格、范围等 meta 信息卡片；只有用户明确要求展示这些信息时，才在标题下方用轻量横向信息条承载。
2. 头图不固定为左右结构。根据内容关系选择承接方式：如果左侧主判断和右侧数字/信息是同一层级对照，可以左右并列；如果是活动信息、嘉宾海报或单一主题介绍，优先使用整宽深色主模块，下方用同级浅色信息块承接时间、地点、数量等信息。
3. 图片类资料不能简单重绘原图。先分析资料类型、信息层级和读图路径，再用 Soft Blocks 重新组织模块；不要复制原图颜色、背景、卡片形态、装饰线和排版节奏。
4. 同一组小模块必须同级处理：如果一组有 8 个指标或要点，8 个都使用相同等级的浅色块面，不要前 4 个彩色、后 4 个白底。
5. 正式 Soft Blocks 输出至少使用 3 种浅色模块背景组合，例如蓝浅底、绿浅底、粉浅底、青浅底、橙浅底、紫浅底；不能只用一种浅色或一组深色块完成全篇。
6. 深色背景模块中的关键数字或关键词统一使用白色；浅色背景模块中的数字或关键词统一使用深色。不要让数字本身承担太多颜色变化。
7. 时间序列、流程步骤、对比组或阶段信息适合放在白色大模块里，用单一强调色表达路径或数据条。
8. Top 列表、任务列表或要点列表适合放在白色大模块里，行间分割线尽量少且非常浅；编号可以用浅灰圆点弱化。
9. 分类条形图同一模块只使用一个主色，轨道条使用该主色的浅色版本。
10. 高光、结论、关键节点优先使用浅色高亮模块；只有重点结论、结语、最终产出或强警示才可以使用深色背景模块。
11. 结构图、组织架构图或流程图如果原图已有清楚的矩阵、树状、泳道、左右对应关系，Soft Blocks 只负责把这些关系翻译成柔彩模块，不要把行列关系拆散成普通卡片流。

## Module Rules

- 颜色优先用于模块底色、状态块和图表条，不用于把每个数字或关键词染成不同颜色。
- 同层级模块的块面、圆角、padding、标题字号、数字字号必须一致。
- 模块内文字必须有清晰的粗细轻重区分：标题、主数据、正文、辅助说明至少要拉开 3 个层级，不要整块文本都用同一种字重。
- 深色模块与浅色模块可以混排，但必须让结构含义清晰：深色用于总览、高光或关键结论，浅色用于普通指标、信息点或辅助说明。
- 深色模块必须克制使用。普通 KPI、趋势、Top 列表、过程拆解、信息点和辅助说明默认使用白色或浅色背景；只有重点结论、结语、最终产出、关键警示等内容才允许使用深色背景。
- 模块内分割线尽量少用；必要时使用极浅色，例如 `rgba(47, 57, 72, .06-.08)`。
- 标题、说明和下方模块网格必须共享同一内容宽度。若用户要求展示 meta 条，meta 条也必须同宽且弱于正文模块。
- 页面背景、模块背景和浅色模块之间必须有明确但柔和的层次，不要再套线框外框。
- 不要在模块里叠多层背景：浅色模块本身已经承载分组，不要再用白色大容器包一组浅色卡；内部条形图的底轨只用同色浅底。
- 首屏 KPI 或关键信息块若与主模块并列，通常直接作为同级浅色块出现，不要放进额外白色面板里；除非它们属于一个明确的表格、矩阵或需要统一边界的主承载模块。
- 模块圆角保持 18px；小型图表条、排名圆点可以使用独立的小圆角。
- 流程、列表或步骤中的数字序号只用于辅助定位，不能比条目标题更抢眼。两位数序号推荐做 48-54px 的小徽章，序号列宽控制在 54-60px；不要使用 80px 以上的大色块占位。序号徽章和右侧标题首行应垂直居中对齐。
- 矩阵型组织关系推荐使用“白色大模块 + 浅色行/列单元”的结构：行列标题、空位占位和撤销/替换关系都要共享同一套列宽。
- 前后更换、左右对照、机构改名等内容推荐使用逐行配对表：左项、中间动作、右项。只高亮真正发生变化的行，避免每行重复强标签。
- 不要把页面做成通用 SaaS dashboard 或营销 landing hero。除非源材料本身要求强品牌首页，否则顶部应以标题、摘要和主数据为主，不能用大面积纯蓝/纯紫横幅压过正文模块。
- 数据复盘页的 KPI 区必须保持主承载模块清晰：标题 + 白色主模块 + 同层级浅色指标项。不要让一组指标变成随机颜色的大卡片拼贴，也不要让色块比数据层级更抢眼。
- 如果同一组指标使用浅色块，浅色可以有轻微变化，但数字颜色必须统一为深色，标签字号、padding、高度和对齐必须一致。
- 趋势图、Top 列表、高光时刻、规律洞察等模块必须复用同一套标题和模块间距；不要每个模块临时发明一种标题条、背景和数字样式。
- 模板规范和已调过的示例是质量下限。生成时如果看起来更像默认 dashboard UI、信息流卡片或营销页，而不是文档可视化长图，需要回到本文件规则重做。

## Data Rules

- 必须遵守 `RULES.md` 的数据真实性和禁止项。
- 数据来源、时间范围、单位和口径必须在顶部说明或模块脚注中明确。
- 只展示源文档给出的事实和可直接计算的派生值；不得编写没有证据的原因解释。
- 颜色只表达视觉分组和模块类型，不暗示源文档没有给出的好坏、风险、因果或策略结论。
- 如果是 Top 列表，只保留一个右侧主数字列，辅助指标放标题下方。
- 条形图、柱状图或进度条如果使用归一化高度或长度，必须标注基准或分母。

## Event Speaker Poster Pattern

- 活动嘉宾长图必须先保留活动标题、时间、地点和简介；这些信息应位于头部，不要被拆成弱 meta 区或放到页面底部。
- 头部活动信息可以采用“整宽深色主模块 + 下方同级浅色信息块”。只有当信息天然左右对照时才使用左右结构。
- 嘉宾列表每张卡必须包含姓名、title、机构/公司和照片；若来源没有提供某项，标注“未提供”，不要补写。
- 嘉宾照片优先下载到本地 assets 再引用；统一使用固定比例容器和 `object-fit: cover`，避免远程资源失效或头像尺寸不一致。
- 同一组嘉宾卡必须等高、同宽、同 padding，照片区域和文字区域在每张卡中保持一致。卡片内部内容顶对齐，不使用 `space-between` 拉开空白。
- 嘉宾卡适合 2 列长图布局；当 title 特别长时，优先调小 title 字号或增加卡片高度，不要压缩照片或让文字溢出。
- 活动页面若有“92 speakers”“3 days”“4 tracks”等总览数字，可以放在头部或底部同级浅色信息块中；数字颜色统一为深色，不随机上色。

## Typography

- 主标题：56-68px，Bold，颜色必须强于或等于模块标题。
- 顶部说明：16-18px，Regular。
- 模块标题：22-28px，Bold。
- 主数据数字：88-120px，Bold。
- KPI 数字：30-42px，Bold。
- 列表主数字：22-28px，Bold。
- 说明 / 口径 / 辅助指标：12-15px，Regular。
- 英文装饰：12-14px，Thin / 300，弱辅助色。

## QA

- 页面是否明显区别于 `minimalist`：应呈现 soft dashboard blocks，而不是讲义或线框表格。
- 顶部标题区是否和页面模块网格同宽。
- 若用户要求自适应或移动端阅读，头图、多列 KPI、嘉宾卡、流程块、结构图和表格必须有明确窄屏重排；仅让 `.page` 变成 `width: 100%` 不算通过。
- 同层级 KPI 是否都使用浅色块面且数字颜色统一。
- 深色模块数字是否统一白色，浅色模块数字是否统一深色。
- 模块内文字是否有标题、主数据、正文、辅助说明的字重区分，而不是整块同粗。
- 同一图表模块内的数据条是否只使用一个主色，底轨是否为对应浅色。
- 模块分割线是否足够少且足够浅。
- 结构图是否保留原图的矩阵、树状、泳道或逐行对应关系；是否没有为了色块风而过度卡片化。
- 数据页是否没有泛 dashboard 化：无过重大色块 hero、无随机彩色 KPI、无多套混杂标题样式。
- 图片类资料是否没有简单重绘原图：没有复制原图配色、背景、卡片形态、装饰线和排版节奏。
- Soft Blocks 是否至少使用 3 种浅色模块背景组合，并以浅色模块为主。
- 深色背景模块是否只用于重点结论、结语、最终产出或关键警示。
- 头图结构是否由内容关系决定，而不是机械左右分栏；是否没有额外白色面板包裹同级浅色块。
- 活动嘉宾页是否保留活动标题、时间、地点、简介和嘉宾照片；嘉宾卡是否等高、照片比例统一、本地资源可加载。
- 是否保留所有关键数字、单位、时间范围和来源。
- 是否没有无证据解释、占位文字、调试文本、截图标注。
