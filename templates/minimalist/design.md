# Minimalist（极简风）

用于发言总结、培训讲义、会议长文提炼、演讲笔记、段落型报告等场景。目标是把长文本整理成清晰、完整、可读性高的 1080px HTML 长图。风格关键词：极简、段落讲义、清晰层级、低装饰、高可读。

## Preview

参考预览图：`assets/styles/minimalist.png`。预览图只用于理解大致风格，不是固定模板。

## When To Use

- 源材料是口语化发言、培训转写、会议记录、长段落笔记，而不是强数据表或复杂架构图。
- 用户希望“清晰、完整、可读性高”，内容需要按主题段落连续阅读。
- 图形不应过度卡片化；更适合极简讲义式段落：一个总纲 + 多个大段落模块。

## Visual System

- 画布：1080px 宽，高度随内容展开。
- 背景：`#F9F9F9`。
- 模块：`#FFFFFF`，不使用描边，用背景、投影和间距区分层级。
- 强调：`#e62828`；柔和强调底：`#ffedeb`。
- 字色：`#333333` / `#666666` / `#999999`。
- 中文：AlibabaPuHuiTi；数字编号：DIN；辅助英文：AlibabaPuHuiTi Thin。
- 字体检查是 Minimalist 的必查项：中文、数字和辅助英文必须分别使用对应字体栈；关键数字、KPI、日期、模块编号、条目编号和图表数值必须明确使用 DIN 或同类数字字体。
- 正文行高：1.5。
- 间距使用 6px 倍数，常用 24px、30px、36px、60px、66px。
- 大标题如果字号超过 48px，行高要适当放松，通常使用 1.18-1.25，避免两行中文和英文混排时显得拥挤。

## Structure

1. 顶部标题区：标题、简短副标题。来源、作者、材料规模、生成方式等元信息默认不展示在页面中；只有用户明确要求时，才用轻量小字或信息栏展示。
2. 核心结论：柔和强调底色，保留左侧红色竖线，和后续普通段落模块形成区分。
3. 主题段落模块：每个模块由标题色条和白底正文组成。
4. 过程/关系模块：可少量使用流程、双栏、编号条目，但不要破坏段落阅读节奏。
5. 来源与待确认：把源文档缺失的信息单独说明，不编造日期、参会人、负责人或截止时间。
6. 底部总结：用深色模块收束全篇。

## Module Rules

- 标题色条：左侧放中文主标题，右侧放模块数字编号和辅助英文。
- 右侧模块数字：只作为定位辅助，必须弱于中文模块标题。可使用 DIN 粗体，但字号、颜色或透明度要克制；不要让 `02` / `03` 这类模块序号成为页面第一视觉。
- 右侧英文：AlibabaPuHuiTi Thin，字重 300，不能抢中文标题层级。
- 数据页模块标题可以采用“左侧中文标题 + 右侧 Thin 英文装饰”的轻量结构。右侧英文只做氛围和扫描辅助，不承载必要信息。
- 正文主体：白底无边框，使用投影区分模块。建议投影 `0 18px 42px rgba(51,51,51,.07)`。
- 段落条目：小红竖条 + 粗标题 + 正文说明。竖条用于扫描，不要把每条都做成独立卡片。红色竖条高度应接近标题首行高度，并与标题首行垂直居中；不要按标题加正文的整体高度居中，避免看起来偏上或偏下。
- 核心结论：保留左侧红色竖线，表示它是总纲。
- 核心结论如果包含一个最重要数字或比例，可以使用“大数字 + 右侧解释”的横向 callout：数字使用 DIN 或同类数字字体放大，右侧保留中文结论说明，中间可用细分割线；不要让关键数字埋在长段落里。
- 顶部元信息如果只是论文标题、作者单位、页数、任务数、来源、类型、风格、范围等背景信息，默认不要放进页面。用户明确要求展示时，不要使用强白卡 + 大投影；优先做成轻量三列/多列信息栏：细分隔线、小标签、较弱正文色；视觉重心应让给后面的核心结论。
- 新增模块必须延续已建立的模块标题样式、分割线、间距和背景规则；例如同级模块都应使用同一套标题字号、右侧辅助说明和标题下分割线，不要单独改成另一种标题层级或过重背景。
- 同一篇长图内优先统一模块标题样式。若标题下方只有一个大内容模块承接，优先使用白底标题 + 黑色下划线的轻量标题样式，让标题和内容成为一个整体；若标题下方有多个并列分模块（例如活动海报里的嘉宾、机构、时间地点等多块内容），可以使用黑底标题条强化分组。
- 标题样式统一不等于内容板式完全重复。长图中如果连续多次出现“红色竖条 + 标题 + 正文”的段落条目，应在不改变字体、颜色和对齐规则的前提下切换承接板式，例如编号论点条、三列浅灰卡、流程块、对比条、限制清单表或结论 callout，避免整篇节奏单一。
- 当同一模块中存在两组关系不同的内容时，不要强行放在同一种灰块网格里。主信息可用浅灰块，补充流程、依赖关系或另一层级信息可用柔和强调底 `#ffedeb` 作为分组背景，但避免使用红色描边框。
- 同一行的并列灰块、白块、流程块、指标块或信息卡必须等高，高度以该组最高内容为准；卡片内部内容统一顶对齐，标题、正文之间的间距由固定 margin/gap 控制，不要使用 `justify-content: space-between` 或垂直居中来填充空白。
- 表格或数值矩阵必须使用显式列宽。每一列的标题和内容应共享同一条左/右对齐线；数字列可以左对齐或右对齐，但同一张表内必须一致，不能出现大面积空白列或列比例失衡。
- 模块内的编号论点条可以使用左侧两位数 + 右侧标题正文，但编号只用于条目定位，不能比条目标题或模块标题更抢眼。建议条目编号比条目标题大 4-8px，并与右侧标题首行垂直居中；不要让编号按整段正文高度居中，也不要让编号显得比模块序号更重要。
- 表格适合使用轻量斑马纹增强扫描：表头保持弱化或白底，数据行可用 `#F3F3F3` / `#F8F8F8` 交替；使用斑马纹后，行 gap 应为 0，用细分割线维持秩序。行内左右 padding 通常为 12-18px。
- 条形图要把名称、条、数值看成一个整体：名称列不要过宽，名称到条形图的 gap 通常 12-18px；底轨在灰底上可用 `#ffedeb`，数值如果表示条形图的百分比或分数，可使用 `#e62828` 与进度条对应。
- 条形图或柱状图外层已有灰底承载时，内部不要再给每一行或图表区套白色背景。层级应服务数据，不要让容器比数值更显眼。
- 若采用黑底模块标题条，标题使用白色，右侧辅助文字使用白色低透明度；黑底标题条只用于同级模块标题，不要把整个内容模块都改成黑底，除非内容资产在黑底上更清晰。
- 同级模块标题条到内容的距离必须等于内容项之间的 gap。常用值为 18px：标题条到首个条目 18px，条目之间 18px，logo 网格横纵间距 18px。
- 大模块之间用 60px 或 66px 拉开；标题条内部、卡片内部、logo 网格内部使用更小间距，避免层级混淆。

## Data Report Pattern

适用于账号复盘、年度数据、业务指标、运营报告等结构化数据长图。

- 先决定主指标，再决定版式。首屏或总览模块应突出 1 个最重要数字，其他指标作为辅助卡片，不要让背景信息抢过主数字。
- 数据页的文案要克制：副标题、来源、口径、说明用 Regular 或 Thin；只有模块标题、关键指标、卡片标题使用 Bold。
- 关键数据、KPI 总览、头图数据必须保持“模块标题 + 白色主模块”的结构。标题可以使用黑色下划线和右侧编号，标题下方应是一个完整白色模块；多个指标在白色模块内部用细分割线或轻量网格组织。不要把每个指标项都做成灰色大块，也不要让灰底铺满整个数据区。
- 任何源文档没有明确给出的“原因、机制、心理、策略效果、未来建议”都不要写成结论。若用户要求洞察，只能基于源表中已经出现的数值、排序和标签描述，不写因果。
- 说明类文字不要做成强模块。口径说明、归一化基准、源表说明和派生公式优先放在图表下方一行常规小字，颜色用 `#666666` 或 `#999999`。
- 同一个数字只出现一次为主。Top 列表里，曝光等主指标可放右侧固定数字列；收藏、评论、点赞、涨粉等辅助指标放标题下方说明，不要再拆出一列造成重复和视觉噪音。
- 同类信息卡必须共享同一套规则。核心数据卡、高光时刻卡等若结构相同，数字字号、标题字号、说明字号、padding 和上下间距应一致；底部红线或浅红背景只表达强调，不改变内部排版规则。
- 高光时刻、日期节点等时间本身是重要信息时，不要把日期弱化成难以阅读的灰色小字。可以把事件名和日期合并成短标题，例如 `单日纪录·2025.11.07`。
- 多个并列总结卡必须等高，内容顶对齐。不要用垂直居中或 `space-between` 填空；若标题被挤成难读的多行，应调整列数或改为纵向列表，而不是压缩文字。
- 对绝对值的月度柱状图，不使用浅色进度底轨。柱高可以按最大值归一化，但必须用口径说明标出归一化基准。

## Event Poster Pattern

适用于活动介绍、嘉宾长图、参会机构展示等场景。

- 活动标题、时间、地点、简介、嘉宾列表和参会机构应尽量放在一个主白色模块中，表示它们属于同一张活动信息海报；不要把时间地点和嘉宾拆成两个互不相关的大模块。
- 人物条目可以使用“左侧方图 + 右侧浅灰信息块”的结构。桌面端左图和右侧信息块应同高、同顶边、同底边；例如头像 `204px × 204px`，右侧信息块也为 204px 高。
- 人物信息块 padding 必须上下对称、左右对称，并使用 6px 倍数；例如 `18px 36px`。
- 人物姓名和序号可以同一行：姓名左对齐，序号右对齐；两者字号一致时序号仍应只作为检索辅助，使用品牌红 `#e62828`，不要再增加额外装饰线。
- title 与机构行属于次级信息，字号应低于姓名；机构行上方分割线使用浅灰，不使用红色，避免误读成重点标注。
- logo 模块必须先确认资源在当前背景上的可读性。若来源 logo 是白色透明版本，不要放浅底；若 logo 是黑色或深色，不要放黑底。必要时换用来源页同组里更适合当前底色的 logo。
- logo 网格适合使用浅灰底单元格承载，每个单元格高度固定，logo 只做等比 contain，不拉伸、不裁切。桌面可用 4 列，手机可降为 2 列。
- 对来源网站资源，应优先下载到本地 assets 再引用，避免预览时依赖远程资源；同时保留真实 alt 文本。

## Content Rules

- 先提炼观点，再放例子。不要把原文流水账整段塞进长图。
- 每个大模块只承载一个主题，例如“培训起点”“为什么会有后端”“思维转变”“案例实践”。
- 每个段落条目包含一个强标题和一段解释，解释尽量保留源文档里的关键例子和术语。
- 没有来源支持的日期、数字、负责人、截止时间、参会人，不要补造；标为“源文档未提供”。

## Double Column Alignment

- 双栏模块如果左右各有相同数量条目，必须按行对齐。
- 不要让左右列表各自自然流式排列，容易出现 A/B/C 与 2xx/4xx/5xx 错行。
- HTML 中优先使用显式行容器，例如每一行一个 `.aligned-row`，内部左右两列分别放对应条目。
- 同级条目使用同一套编号形态：A/B/C 或 01/02/03。
- 左右列宽使用 `grid-template-columns: 1fr 1fr`，列间距使用 24px 或 30px。

## QA

- 1080px 页面宽度稳定。
- 无横向溢出。
- 若用户要求自适应或移动端阅读，多列、双栏、表格、流程块和关键数字 callout 必须有明确窄屏重排；仅让 `.page` 变成 `width: 100%` 不算通过。
- 大模块宽度一致。
- 必查字体：中文是否使用 AlibabaPuHuiTi，辅助英文是否使用 AlibabaPuHuiTi Thin / 300，数字是否使用 DIN 或同类数字字体。
- 必查数字字体：主数据、KPI、日期、图表刻度/数值、模块编号和条目编号是否都套用数字字体栈；不能只依赖页面默认中文字体。
- 辅助英文使用 Thin。
- 双栏同级条目逐行对齐。
- 核心结论和普通模块有清晰区分。
- 低优先级元信息没有抢过核心结论。
- 模块右侧序号是否弱于中文模块标题，没有成为第一视觉。
- 段落条目的红色竖线是否与标题首行垂直居中，且竖线高度接近标题首行高度。
- 编号论点条的编号是否只作为辅助定位，并与标题首行垂直居中；编号不能抢过条目标题或模块标题。
- 表格斑马纹连续、无行间白缝，列对齐清楚。
- 关键数据区保持白色主模块承载，指标项没有被大面积灰底拆散。
- 条形图底轨清晰，名称、进度条和数值的距离紧凑一致。
- 无调试标记、占位文本、红框标注或 TODO。
