# Document Visualizer

一个文档可视化设计系统 skill：先把源材料整理成可信的信息模型，再从风格目录中选择合适的视觉模板，产出清晰、完整、可读性高的 SVG 或 HTML 图文排版。

这不是固定模板生成器。`CATALOG.md` 用来选择版式/风格，`RULES.md` 给出所有输出必须遵守的硬规则，`templates/<slug>/design.md` 提供具体风格的视觉规范。

## 使用时机

- 用户给出文档、笔记、会议纪要、发言转写、论文、技术方案、PRD、源码分析、结构化数据、结构图或流程图，并希望整理成图文视觉。
- 用户明确说使用 `document-visualizer`。
- 用户需要长图、文档配图、技术方案可视化、论文讲解、会议纪要整理、结构图重绘、流程图重绘、PPT 页面或 HTML 可视化页面。

## 工作流程

1. **读取源材料。**
   - 识别文件类型和可用内容。
   - 提取标题层级、章节、表格、图示、日期、人物、实体、指标、决策、风险和待办。
   - 如果源材料是图片结构图或流程图，先提取可识别的节点、连线、方向、分组、层级和疑似文字。
   - 如果内容提取不完整，先说明缺失部分，再进入设计。

2. **建立信息模型。**
   - 生成简洁的文档大纲。
   - 区分事实、分析、假设和待确认问题。
   - 保留源文档中的关键数字、标签、日期和原始措辞。
   - 不编造数据；没有来源支撑的推断必须标为假设。
   - 对 OCR 不确定、关系不清或被遮挡的图中内容，标为“疑似”或“未识别”，不要自行补全。

3. **选择视觉模式和风格。**
   - 阅读 [`RULES.md`](RULES.md)，了解信息完整性、版式、字体、颜色和 QA 的硬规则。
   - 阅读 [`CATALOG.md`](CATALOG.md)，根据源材料结构和用户目标推荐并选择一个风格。
   - 如果用户没有指定风格，先按文档类型给出一句简短推荐理由，然后直接使用最合适的风格继续制作；不要停下来让用户二选一，除非多个风格同样合适且会明显影响结果。
   - 只打开选中的一个 [`templates/<slug>/design.md`](templates/)；不要为了比较同时打开多个风格模板。
   - 如果用户已经指定风格或参考样式，优先尊重用户指定。

4. **制作输出。**
   - 按所选模板组织版式，但不要照搬示例内容。
   - 当 SVG 难以稳定处理长文本换行或复杂预览时，优先使用 HTML/CSS。
   - 如果用户要求 PPT，输出适合幻灯片尺寸的 SVG/HTML 页面，或使用当前环境可用的演示文稿工具。
   - 核心文字尽量保留为真实文本，不转成路径或图片。

5. **验证并迭代。**
   - 按 [`RULES.md`](RULES.md) 的 QA 清单检查输出。
   - SVG 输出可运行 [`scripts/svg_check.py`](scripts/svg_check.py)。
   - HTML 输出优先用浏览器检查宽度、溢出、模块对齐、文字层级和占位内容。
   - 用户明确要求“先不要渲染 PNG”或“先直接出 HTML”时，不要生成最终图片。

6. **交付。**
   - 说明源文档结构摘要、推荐/选择的视觉策略、生成的文件路径、关键假设/缺失信息、已执行的验证。
   - 如果是可切换风格的内容，告诉用户后续可以换用 catalogue 里的其他风格重新渲染。

## 文件

- [`RULES.md`](RULES.md)：所有文档可视化输出的硬规则和 QA 清单。每次使用都要读。
- [`CATALOG.md`](CATALOG.md)：版式/风格目录。用它选择一个合适模板。
- [`templates/<slug>/design.md`](templates/)：每个风格自己的视觉规范和适用场景。
- [`scripts/svg_check.py`](scripts/svg_check.py)：基础 SVG 检查脚本。
